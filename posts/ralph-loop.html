I Let an AI Code for 36 Hours Straight. Here's the System That Made It Work.

March 21, 2026 · Marcus Vorwaller

Last week I decided to build a Figma clone. Not a toy demo—a real design tool. Canvas with pan/zoom. Layers with search, keyboard nav, drag reorder. Auto-layout with flexbox-style distribution. Frames with constraints and device presets. A pen tool. Gradient fills and drop shadows. Animation timelines with keyframes. Branching and version history. Real-time collaboration with cursors. A plugin system with a sandboxed runtime. An AI chat panel that can modify the canvas directly.

I built it inside Perch, a dev dashboard I've been working on. Google had just released updates to Stitch that I wanted to try out, and since theirs isn't open source, I figured I'd just build my own.

By the time I'm writing this, there are 93 source files in the designer module, over 51,000 lines of code, and 244 commits in the last 48 hours. I wrote almost none of it by hand. A Claude agent has been running in a loop for 36+ hours, picking up tasks, implementing them, committing, and moving on.

Not another loop post

The "Ralph loop" pattern—named after Ralph Wiggum, because the agent just keeps going regardless—has been around since mid-2025. Geoffrey Huntley popularized it: put a coding CLI in a while true loop, point it at a plan, let it run. There are several open source implementations now. The basic loop is simple, and a lot has been written about it.

I want to talk about the system around the loop, because the loop itself isn't the interesting part. The interesting part is what makes 36 hours of autonomous coding produce good code instead of a mess.

The setup

I have a task manager called td that I built specifically for AI agents. It tracks issues, priorities, sessions, and reviews. The key feature: agents can implement work, but they can't review their own work. A different session has to do the review.

Here's how the designer project started:

  1. I wrote a high-level plan. "Build a Figma-class design tool. Here are the major feature areas." A planner tool broke this into concrete goals.
  2. I created an epic in td and decomposed the plan into ~90 individual tasks, each scoped to one feature. Layer panel keyboard navigation. Image element support. Auto-layout engine. Canvas version history. Each one small enough for a single agent session.
  3. I started the Ralph loop: a bash script that runs claude in print mode, once per iteration, with a prompt that says "check td, pick the highest-priority open task, implement it, commit, and exit."

Then I went to bed.

Three things that make it work

The task queue is alive

Most Ralph loop setups work from a static PRD or a markdown checklist. Mine uses td as a living backlog. While the agent is grinding through P2 tasks at 3 AM, I can pick up my phone and add a new task to the epic. "Canvas doesn't render on mobile Safari." Done. It gets added at P3 and the loop picks it up when the higher priority stuff clears. I don't have to restart anything or think about scheduling. I just throw work into the queue.

Independent review catches real problems

td enforces that the implementing agent can't close its own review. On iteration N, the agent builds the layer panel search feature and submits it for review. On iteration N+1, a fresh agent instance—with no context from the previous session—reviews it. Different eyes every time. Not "did the tests pass" but an actual code review with a clean perspective. It catches over-engineering, missing edge cases, and style inconsistencies that the implementing agent was blind to.

Context isolation keeps quality high

Each agent iteration starts from scratch with context only for one task. After 36 hours of continuous work, there's no accumulated context window bloat, no confusion about which file it was editing three features ago. Iteration 200 is as sharp as iteration 1. Most people who try long-running agents hit a wall where quality degrades as context fills up. With one task per iteration, that never happens.

What doesn't work

It's not magic. Some tasks come back from review with real issues and need a second pass. Occasionally the agent misunderstands a task and builds something adjacent to what I wanted. Complex UI interactions—things that require visual judgment—are harder for the agent than pure logic.

There's a 4,196-line god component that's a testament to what happens when you let an agent keep adding features to the same file without refactoring. (There's already a task in the queue to break it up. The loop will get to it.)

And I've had to restart the loop a few times due to my own mistakes—usually messing up the script or the prompt.

Why this matters

There's a lot of discourse about AI coding that falls into two camps: "AI will replace all programmers" and "AI code is garbage." Neither is right. The more interesting truth is in the middle. AI agents are good at implementing well-scoped features when you give them clear context and a way to get feedback. They're bad at architecture, taste, and knowing when to stop. The system around the agent—the task management, the review process, the queue—is what determines whether you get good output or garbage.

I'm not building a startup around this. I built the Figma clone because Google released Stitch updates that inspired me, theirs isn't open source, and I had ideas I wanted to try. The loop let me go from idea to working implementation in a weekend.

That's the part I keep coming back to. Not the technology, but the feeling: "I wonder if I could build a design tool" on Friday night, and by Sunday afternoon, I have one. The system made it possible. The loop is just a while true. The system is what made it work.

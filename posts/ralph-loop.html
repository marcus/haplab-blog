The Backlog Loop

March 22, 2026 · Marcus Vorwaller

For the past few months I've been experimenting with different ways of working with agents — that's what led to projects like sidecar and td, plus a bunch of other stuff I haven't open sourced yet. This weekend I stumbled into a way of working that I'm finding really nice, and it feels like it's pointing toward something.

I need a good name for it. "Continuous development" is close but not quite right. "Autopilot" is too vague. I'll figure it out — for now I'm calling it the backlog loop.

Where this came from

I'd been playing with Ralph loops — the idea of running agents in a loop with minimal intervention. The concept is great but in practice I found it a bit too loose and non-deterministic for my taste. This is a variation that added more structure and, for me at least, works a lot better.

The basic idea: I use td, my task manager, to create an epic and add stories to it. Then I start a loop that works through those stories one at a time. While work is only happening sequentially — not in parallel — it still seems to come together at roughly the same speed, and it completely eliminates all the complexity of worktrees, multiple branches, and merging. I'll take that trade every time.

My job becomes managing the queue. The agent just works.

How the loop works

Each loop iteration is a fresh agent session — no memory of what happened before. When an agent starts, it:

  1. Checks if there are any tasks in review (up to three) and reviews them, flagging issues or creating new tickets if something needs follow-up
  2. Picks up the highest-priority open task and works on it
  3. Provides two forms of proof before marking done: a passing test run and screenshots of the feature actually working

Then the next session starts and does the same thing. That review step matters a lot — an agent can't approve its own work. A fresh session has to look at it with clean eyes.

The other key insight is that fresh sessions keep the context window from getting bloated. Long-running sessions start to drift. Each task getting its own session means each task gets focused attention.

One other thing worth mentioning: tasks are worked in priority order. So if I think of something urgent, I can just create a high-priority ticket and it'll float to the top of the queue on the next iteration. I don't need to stop anything or reorganize — it just happens. And since I'm chatting with my OpenClaw agent to create and refine tasks anyway, adjusting the queue is as easy as saying "actually skip that one" or "let's break this into smaller pieces."

For this to work, the codebase needs to be organized in a way that agents can navigate. Skills files, clear conventions, predictable directory structure. There's some upfront work to get that in place. But once it's there, the loop is remarkably simple — it doesn't need babysitting.

What it built over the weekend

I proved this out by creating a plan for a Figma-style design tool — Designer — that lives inside a larger dev dashboard project I've been building. The initial plan had around 60–70 tasks. I kicked it off Friday night and woke up Saturday to what is, by any measure, a solid vibe-coded design tool.

I kept adding tasks over the weekend. Generally I found I was always a few tasks ahead of the agent, which is exactly the right balance — I can walk away from my laptop and work is still getting done.

Here's what Designer has after all that:

  • Canvas — pan and zoom, viewport culling, LOD rendering, dirty-region tracking, minimap, rulers, and guides with on-canvas numeric editing
  • Elements — rectangles, ellipses, lines, arrows, paths, text, groups, frames, images. Multiple fills and strokes per element. Blend modes (16 of them). Drop shadow, inner shadow, blur, and background blur
  • Text — rich text editing with mixed styles per character. Google Fonts integration with a full font picker (search, categories, preview, favorites). Variable font axes and OpenType feature controls
  • Frames — auto-layout with direction, gap, padding, wrap, and constraints. Device presets and artboard templates with a gallery. Clip content toggle
  • Layers panel — search with name filter and type chips. Full keyboard navigation. Batch visibility and lock toggling. Drag reorder and nesting. Scroll to selected
  • Tools — Select, Rectangle, Ellipse, Frame, Line, Pencil (with bezier fitting and pressure sensitivity), Pen (with bezier curve editing), Text, Hand, Eyedropper, and a Measure tool
  • Styling — full color picker (HSV field, hue/opacity sliders, hex/RGB/HSL inputs). Gradient editor. Copy/paste styles between elements
  • Vector editing — boolean operations (union, subtract, intersect, exclude). Vector and alpha masks. Enhanced pencil tool with eraser mode
  • Components — component library panel with search, drag-to-canvas, instance actions. Design token system with CRUD, CSS/JSON export, style picker, bulk apply/detach
  • Collaboration — real-time cursors and remote selections over WebSocket
  • Versioning — version history timeline with search, thumbnails, diffs, and preview mode. Branch and merge for canvas versions
  • AI integration — chat panel that creates and modifies canvas elements directly. Click-to-reference elements in chat. Visual feedback for agent operations with pulsing highlights and auto-pan. AI image generation (OpenAI + Gemini). Generative layout with prompt starters
  • Developer handoff — inspect mode with CSS generation and red-line measurements
  • Design critique — WCAG contrast checking, spacing, alignment, font size, and touch target audits
  • Animation — timeline panel, keyframe editor, property interpolation
  • Prototyping — prototype viewer, click targets, flow diagrams
  • Plugins — sandboxed runtime, API bridge, SDK types
  • Export — PNG, SVG, and PDF with scale and format options
  • Mobile — iOS Safari canvas rendering fixes, touch-action support
  • Keyboard shortcuts — organized by category with a shortcuts overlay. Z-order operations, smart selection (select all similar), zoom presets

78 component files. About 50,000 lines of Svelte and TypeScript. The perch repo has 1,062 commits total, the majority from the loop running over the past few weeks. There's a test suite — a mix of unit tests and Playwright e2e specs. Honestly the coverage isn't where I'd want it for a production app, but it exists and the agents use it. Something to improve next time around.

I didn't write any of that code. I haven't read most of it. That's kind of wild to say out loud.

The dashboard

I also built a little dashboard so I can see what the loop is up to — recently completed tasks, how long each one took, what's coming up next. It's a Python Rich TUI that tails the td log for a given epic. When I come back to my laptop after a few hours away, I can see exactly what happened.

What makes this feel different

I've been building software for a long time. This feels like something shifted.

It's not that the AI is doing magic. The individual tasks still require good tickets — "make the app better" isn't going to produce great results. The quality of output tracks closely to the quality of what goes into the backlog. That part hasn't changed.

What's changed is the rhythm. Development is now happening as fast as I come up with ideas and write them down. There's no gap between "I want this feature" and "work has started on this feature." I add a ticket, and the next time the loop picks up, that ticket gets worked on. I can be out walking and think of something, jot it down on my phone, and by the time I'm back at my desk it might already be done.

The job description has shifted. For Designer, what I'm actually doing is: thinking about what the software should be, writing tickets clear enough for an agent to implement, and occasionally reviewing what came back. That's it. There's no "sit down and implement this" step.

I'm going to keep running with this. The tools I used are open source if you want to try something similar:

  • td — task manager built for AI agents (the backlog engine)
  • sidecar — dev dashboard that shows what's happening

And if you've got a good name for this workflow, I'm all ears.

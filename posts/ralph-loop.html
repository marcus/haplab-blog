Vibe Coding Doesn't Have to Mean Bad Software

March 21, 2026 · Marcus Vorwaller

I built a Figma clone over a weekend. I didn't write any of the code, and I've never read it. A Claude agent ran in a loop for about a day and a half, and by the end there were 93 source files, 51,000 lines of code, and 244 commits in the repo.

This is vibe coded software. I'm not going to pretend otherwise. But I think it's also an example of how vibe coding doesn't have to produce throwaway garbage, and that has some real implications for the industry.

Why a Figma clone

Google released updates to Stitch that I wanted to play with. Stitch isn't open source, so I decided to build my own version inside Perch, a dev dashboard I've been working on. It started as a weekend experiment and turned into something I'm going to keep developing.

The Ralph loop

The "Ralph loop" has been around since mid-2025. Geoffrey Huntley popularized it: put a coding agent in a while true loop, point it at a plan, let it run. Named after Ralph Wiggum because the agent just keeps going regardless. There are several open source versions and the basic loop isn't complicated.

What I want to write about is the scaffolding around my version of it. The loop alone can produce decent results, but without visibility into what's happening and the ability to steer it, you're flying blind. My setup gives me both, and the combination of a solid upfront plan with continuous observability is what I think made the output as good as it is.

How I set it up

I have a task manager called td that I built for AI agents. It tracks issues, priorities, and reviews. The important thing about td is that agents can implement work, but they can't review their own work. A different session has to do that.

I wrote a high-level plan, broke it into about 90 tasks under a single epic, and scoped each task to a single feature. Layer panel keyboard navigation. Image element support. Auto-layout engine. Canvas version history. Each one was sized so a single agent session could complete it without running out of context.

The loop itself is a bash script. It runs claude in print mode, one iteration at a time. Each iteration checks td, picks the highest-priority open task, implements it, commits, and exits. The script handles rate limit backoff, a 45-minute timeout per iteration, and a fallback model if the primary is overloaded.

I started it Friday night and went to bed.

What made the difference

A live task queue instead of a static checklist. Most Ralph loops work from a PRD or a markdown file. Mine uses td as a backlog that I can add to at any time. The agent was working through P2 tasks at 3 AM. I woke up, thought of something on my phone, and filed a P3 bug. The loop picked it up later that day without me restarting anything.

Forced independent review. On iteration N, the agent builds the layer panel search feature and submits it for review. On iteration N+1, a brand new agent instance with zero context from the last session reviews it. Fresh eyes every time. Not "do the tests pass" but an actual code review from a completely clean perspective. It catches things the implementing agent can't see because it was too close to the work.

One task per iteration. Each agent starts with context for exactly one task. There's no accumulated bloat from prior iterations and no confusion about which file it was editing three features ago. Iteration 200 has the same clarity as iteration 1. People who try long-running agents usually see quality degrade as the context window fills up. This approach sidesteps that completely.

What went wrong

Some tasks came back from review with real problems and needed a second pass. The agent occasionally misunderstood a task and built something adjacent to what I asked for. Anything requiring visual judgment was harder than pure logic work.

There's a 4,196-line component file that shows what happens when an agent keeps adding features to the same file without stepping back to refactor. There's already a task in the queue to break it apart.

I also had to restart the loop a few times because I messed up the script or the prompt.

What this means

I've never read any of this code. That's the part I keep coming back to. Fifty completed features, 51,000 lines, a test suite, and I couldn't tell you what any specific function does. The system I built around the agent is what kept the quality bar up: the upfront planning, the forced independent review, the one-task-at-a-time isolation.

That says something about where software is headed. The discourse right now tends to split into "AI will replace all programmers" and "AI code is garbage." Both are wrong. Agents can build real software when you give them structure, context, and feedback. They can't do architecture or make taste decisions. The system around the agent is what determines whether the output is usable or not.

For the software industry, I think this is a preview. Not of programmers becoming obsolete, but of the job changing. I spent my weekend on planning, task design, and system architecture. The code wrote itself. That ratio is only going to shift further.

I plan to open source all of this. Some of it already is:

  • td — the task manager that runs the whole workflow
  • Perch — the dev dashboard where the designer lives
  • marcus-skills — the Ralph loop scripts and TUI dashboard

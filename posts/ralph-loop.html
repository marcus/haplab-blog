I Let an AI Code for 36 Hours Straight

March 21, 2026 · Marcus Vorwaller

Friday night I decided to build a Figma clone. By Sunday afternoon I had one.

93 source files. 51,000 lines of code. 244 commits. I wrote almost none of it by hand. A Claude agent ran in a loop for a day and a half while I mostly did other things.

I should back up.

Why a Figma clone

Google released updates to Stitch that I wanted to play with. Stitch isn't open source, so I decided to build my own version inside Perch, a dev dashboard I've been working on. This started as "let me see how far I can take this over a weekend" and turned into something that actually works.

The Ralph loop

The "Ralph loop" has been around since mid-2025. Geoffrey Huntley popularized it: put a coding agent in a while true loop, point it at a plan, let it run. Named after Ralph Wiggum because the agent just keeps going regardless. There are several open source versions. The basic loop isn't complicated.

What I want to write about is everything around the loop, because the loop by itself produces mediocre output. The scaffolding is what made this work.

How I set it up

I have a task manager called td that I built for AI agents. It tracks issues, priorities, and reviews. One rule: agents can implement work, but they can't review their own work. A different session has to do that.

I wrote a high-level plan, broke it into about 90 tasks under a single epic, and scoped each one to a single feature. Layer panel keyboard navigation. Image element support. Auto-layout engine. Canvas version history. Each task small enough for one agent session.

The loop is a bash script. It runs claude in print mode, one iteration at a time. Each iteration: check td, pick the highest-priority open task, implement it, commit, exit. Script handles rate limit backoff, a 45-minute timeout per iteration, and a fallback model if the primary is overloaded.

I started it and went to bed.

What made the difference

A live task queue instead of a static checklist. Most Ralph loops work from a PRD or a markdown file. Mine uses td as a backlog that I can add to at any time. The agent was grinding through P2 tasks at 3 AM. I woke up, thought of something on my phone, filed a P3 bug. The loop picked it up later that day. I never restarted anything.

Forced independent review. On iteration N, the agent builds the layer panel search. It submits the work for review. On iteration N+1, a brand new agent instance with zero context from the last session reviews it. Fresh eyes every time. Not "do the tests pass" but actual code review. It catches stuff the implementing agent can't see.

One task per iteration. Each agent starts with context for exactly one task. No accumulated bloat. No confusion about which file it was editing three features ago. Iteration 200 is as sharp as iteration 1. People who try long-running agents usually hit a wall where quality drops as context fills up. This sidesteps it completely.

What went wrong

Plenty. Some tasks came back from review with real problems. The agent occasionally misunderstood a task and built something adjacent to what I asked for. Anything requiring visual judgment was harder than pure logic work.

There's a 4,196-line component file that's what happens when you let an agent add features to the same file without refactoring. There's already a task in the queue to break it apart.

I also had to restart the loop a few times because I messed up the script or the prompt. Operator error.

What I keep thinking about

Friday night I had an idea. Sunday afternoon I had a working design tool with collaboration, version branching, animation, and a plugin system. I was outside for most of it.

The agent didn't do the hard part. I did the hard part weeks ago when I built td, wrote the review system, set up the loop infrastructure, and figured out how to scope tasks so an agent could complete them cleanly. The weekend was just the payoff.

There's a lot of "AI will replace programmers" and "AI code is garbage" going around. Both miss the point. Agents are good at building well-scoped features when you give them context and feedback. They're bad at architecture and taste. The system you build around the agent determines what comes out the other end.

I plan to open source all of this. Some of it already is:

  • td — the task manager that runs the whole workflow
  • Perch — the dev dashboard where the designer lives
  • marcus-skills — the Ralph loop scripts and TUI dashboard

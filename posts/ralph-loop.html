The Continuous Developer

March 22, 2026 · Marcus Vorwaller

I have a design tool called Designer that I've been building inside a larger dev dashboard project. Over the past week or so, I've had a coding agent running in a loop against the codebase, picking up tasks from a backlog, implementing them, committing, and moving on. When I add a ticket, it gets worked on. When there's nothing in the queue, the loop waits. I don't start sessions or type prompts. I just write tickets.

The loop has been running more or less continuously. It's produced 93 source files, about 51,000 lines of Svelte and TypeScript, and 244 commits. I didn't write any of that code and I haven't read it. It was all built by Claude sessions working one task at a time off a shared backlog.

I think there's something interesting here beyond the output. It's starting to feel less like "using AI to code" and more like a different way of working entirely.

How it works

I have a task manager called td that I built for AI agents. It tracks issues, priorities, and code reviews. When an agent finishes a piece of work, it can't approve its own review — a different agent session has to look at it. That happens naturally because each loop iteration is a fresh session with no memory of the last one.

The loop is a bash script. Each iteration picks the highest-priority open task, builds it, commits, and exits. The script handles rate limits, timeouts, and model fallbacks. I started it with about 90 tasks in the backlog, but I've been adding new ones as I think of them. The loop doesn't care when tasks were added or by whom. It just works the queue.

The codebase itself is organized so that agents can navigate it. There are skills files that describe patterns, conventions, and how things connect. Tests cover enough surface area that the agent can verify its own work before submitting for review. The directory structure is predictable. None of this was done specifically for the loop — it's just how I've been building things for a while now — but it turns out that a codebase organized for agents is also a codebase that can be developed continuously.

The shift

The part I keep thinking about is that the loop doesn't need me to do anything except add tickets. I don't open an IDE. I don't start a Claude session. I don't write a prompt. I think about what the software needs, write a ticket describing it, set a priority, and drop it in the backlog. It gets picked up and built.

Sometimes I write tickets on my phone while I'm out walking. Sometimes I think of something over coffee and file it before I forget. The loop is always there, working through whatever's at the top of the queue. When I check in later, the feature exists.

This is starting to feel like the job description is changing. For this project, what I do is:

  • Plan what the software should be
  • Write tickets that describe the work clearly enough for an agent to complete
  • Review what was built (or let the next loop iteration review it)
  • Occasionally add bugs or improvements when I notice something

That's it. There's no coding step. There's no "sit down and implement this" step. The backlog is the interface to the codebase.

What makes a codebase work this way

Not every codebase could run like this. The ones that can tend to have a few things in common:

The structure is legible. An agent landing in the repo for the first time can figure out where things go and how they connect. Consistent naming, a clear directory layout, and documentation that describes the architecture all help. If a senior engineer would need a week to get oriented, an agent will struggle too.

There's a test suite that the agent can run. It doesn't need to be comprehensive, but it needs to cover enough that the agent can tell whether it broke something. Agents are good at writing new tests for the features they build, but they need an existing suite to run against.

The tickets are specific. "Make the app faster" won't produce good results. "Add viewport culling to the canvas renderer so elements outside the visible area aren't drawn" will. The quality of the output tracks directly to the quality of the ticket. This is where the human work has moved — from writing code to writing good descriptions of what the code should do.

And there's a review mechanism. Agents make mistakes. They over-engineer things, miss edge cases, and sometimes misunderstand the task entirely. Having every piece of work reviewed by a separate session with fresh context catches a lot of this before it compounds.

Where I think this goes

I've been a developer for over 20 years and a manager for the last several. The feeling of working this way is different from anything I've experienced before. It's not pair programming. It's not delegating to a junior developer. It's closer to being a product owner who happens to have an engineering team that works 24 hours a day and never needs a standup.

I think as more codebases get organized to work well with agents — legible structure, good tests, clear documentation — this kind of continuous development is going to become common. The loop technology is simple. The hard part is making the codebase ready for it, and that's mostly just good engineering practices that we should have been doing anyway.

A lot of people are working this way already or starting to. Things are moving fast and I'm curious to see where it all goes.

Vibe Coding Doesn't Have to Mean Bad Software

March 21, 2026 · Marcus Vorwaller

I built a Figma clone over a weekend. I didn't write any of the code, and I've never read it. A Claude agent ran in a loop for about a day and a half, and by the end there were 93 source files, 51,000 lines of code, and 244 commits in the repo.

This is vibe coded software. I'm not going to pretend otherwise. But I think it's also an example of how vibe coding doesn't have to produce throwaway garbage, and that has some real implications for the industry.

Why a Figma clone

Google released updates to Stitch that I wanted to play with. Stitch isn't open source, so I decided to build my own version inside Designer, a design tool I've been building as part of a dev dashboard project. It started as a weekend experiment and turned into something I'm going to keep developing.

The Ralph loop

The "Ralph loop" has been around since mid-2025. Geoffrey Huntley popularized it: put a coding agent in a while true loop, point it at a plan, let it run. Named after Ralph Wiggum because the agent just keeps going regardless. There are several open source versions and the basic loop isn't complicated.

What I want to write about is the scaffolding around my version of it. The loop alone can produce decent results, but without visibility into what's happening and the ability to steer it, you're flying blind. My setup gives me both, and the combination of a solid upfront plan with continuous observability is what I think made the output as good as it is.

How I set it up

I have a task manager called td that I built for AI agents. It tracks issues, priorities, and reviews. The important thing about td is that agents can implement work, but they can't review their own work. A different session has to do that.

I wrote a high-level plan, broke it into about 90 tasks under a single epic, and scoped each task to a single feature. Things like layer panel keyboard navigation, image element support, the auto-layout engine, and canvas version history. Each task was sized so a single agent session could complete it without running out of context.

The loop itself is a bash script. It runs claude in print mode, one iteration at a time. Each iteration checks td, picks the highest-priority open task, implements it, commits, and exits. The script handles rate limit backoff, a 45-minute timeout per iteration, and a fallback model if the primary is overloaded.

I started it Friday night and went to bed.

Three things that helped

A live task queue instead of a static checklist. Most Ralph loops work from a PRD or a markdown file. Mine uses td as a backlog that I can add to at any time. I like that I can think of something at any hour, file it on my phone, and know the loop will pick it up when the priority is right. I don't have to restart anything or coordinate with the running agent. I just add work to the queue and it gets done.

Forced independent review. On iteration N, the agent builds the layer panel search feature and submits it for review. On iteration N+1, a brand new agent instance with zero context from the last session looks at what was built. It catches things the implementing agent can't see because it was too close to the work, and the review is a real code review, not just a test pass check.

Observability and steering. There's a TUI dashboard that shows task counts, recent commits, velocity, and a scrollable log of what the current iteration is doing. I can see what's happening without interrupting it. And because the task queue is live, I can reprioritize or add work at any point. The loop keeps running and adjusts to whatever's at the top of the backlog. It's less like launching a rocket and more like driving a car.

Where it fell down

Some tasks came back from review with real problems and needed a second pass. The agent occasionally misunderstood a task and built something adjacent to what I asked for. Anything requiring visual judgment was harder than pure logic work.

There's a 4,196-line component file that shows what happens when an agent keeps adding features to the same file without stepping back to refactor. There's already a task in the queue to break it apart.

I also had to restart the loop a few times because I messed up the script or the prompt.

Why I think this matters for the industry

I've never read any of this code. Fifty completed features, 51,000 lines, a passing test suite, and I couldn't tell you what any specific function does. The system I built around the agent is what kept the quality bar up: the upfront planning, the forced independent review, and the ability to steer the work while it's running.

The discourse right now tends to split into "AI will replace all programmers" and "AI code is garbage." Both are wrong. Agents can build real software when you give them structure, context, and feedback. They can't do architecture or make taste decisions. The system around the agent is what determines whether the output is usable or not.

I spent my weekend on planning, task design, and system architecture. The code wrote itself. That ratio is only going to shift further, and I think that's a preview of where the job is headed for a lot of us.

I plan to open source all of this. Some of it already is:

  • td — the task manager that runs the whole workflow
  • Designer — the design tool (part of a larger dev dashboard)
  • marcus-skills — the Ralph loop scripts and TUI dashboard
